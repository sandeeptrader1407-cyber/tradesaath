#### F-010: Razorpay webhook signature compared with `!==` (timing-unsafe)
**Severity:** MEDIUM
**Section:** 2
**Location:** `app/api/webhooks/razorpay/route.ts:30`
**What I found:** Signature verification uses plain string comparison: `if (expectedSignature !== signature)`. JavaScript `!==` short-circuits on first byte difference, leaking timing information.
**Why it matters:** A motivated attacker can iterate byte-by-byte to forge a valid signature. Realistically rare for HMAC-SHA256 over a short hex string and not currently exploitable through Vercel's request lifecycle, but it's the textbook crypto-101 mistake and trivially fixable.
**Suggested fix:**
- Convert both to `Buffer` and compare with `crypto.timingSafeEqual(Buffer.from(expectedSignature, 'hex'), Buffer.from(signature, 'hex'))`.
- Wrap in try/catch — `timingSafeEqual` throws if buffer lengths differ.
**Effort:** S (< 2h)

#### F-012: `/api/extract` is public AND calls Claude API on intake fallback
**Severity:** HIGH
**Section:** 2 (auth surface) / Section 8 (cost)
**Location:** `middleware.ts:18` (public list); `app/api/extract/route.ts:67-83`
**What I found:** Middleware whitelists `/api/extract` as public. The handler tries the local intake pipeline first; if it returns 0 trades, it falls back to a Claude `messages.create` call with the file content (PDF base64, image base64, or CSV/Excel text). Rate limit is 5 requests per **IP** per 15 min; there is a `DISABLE_AI_ANALYSIS` kill switch and a 10 MB file cap. There is **no `userId` check** anywhere in the route.
**Why it matters:** This is the "free first analysis" path Sandeep designed (per product vision: free tier = local + minimal AI to impress). It's intentionally public, so the threat model is cost-abuse, not data theft. Per-IP rate limiting is fragile against (a) mobile carrier NAT (legitimate users sharing one IP), (b) attackers behind residential proxy networks. A 10 MB PDF base64'd in a Claude request can burn 50K+ input tokens; 5 reqs × continuously rotated IPs = real money.
**Suggested fix:**
- Keep the route public (product decision) but add a SECOND gate: require an `anon_id` cookie, and rate-limit `extract:${anonId}` at e.g. 1/day in addition to the IP limit. Anon cookies survive across navigation and force a tiny cost on attackers.
- Add a daily global ceiling: `extract:global:${YYYYMMDD}` capped at e.g. 500 calls/day. If exceeded, return `503 AI_DISABLED`. This is your circuit breaker.
- Log every Claude fallback with `{ip, anonId, fileSize, tokens}` so you can spot abuse.
**Effort:** M (2-8h)

#### F-015: Multiple RLS policies use `USING (true)` / `WITH CHECK (true)` — universal access via the public anon key
**Severity:** BLOCKER
**Section:** 3
**Location:**
- `supabase/migrations/002_add_kpis_and_user_plans.sql:43` — `user_plans` "Service can manage all plans" `FOR ALL USING (true)`
- `supabase/migrations/002_add_kpis_and_user_plans.sql:54` — `trade_sessions` "Service can manage all sessions" `FOR ALL USING (true)`
- `supabase/migrations/006_coupons.sql:31-37` — `coupons` and `coupon_redemptions` `FOR ALL USING (TRUE)`
- `supabase/migrations/003_user_journeys.sql:18` — `user_journeys` `FOR ALL USING (true)`
- `supabase/migrations/001_trade_sessions.sql:23-25` — `trade_sessions` "Allow insert" `FOR INSERT WITH CHECK (true)`
- `supabase/migrations/003_trade_analysis.sql:36` — `trade_analysis` "Service can insert trade analysis" `FOR INSERT WITH CHECK (true)`
- `supabase/migrations/004_raw_files.sql:33` — `raw_files` "Users can insert files" `FOR INSERT WITH CHECK (true)`
**What I found:** Multiple policies are written without a role qualifier (`TO service_role`) or any predicate. Supabase applies these to **all roles**, including the `anon` role used with the public `NEXT_PUBLIC_SUPABASE_ANON_KEY`. Although the codebase deliberately routes everything through API routes that use the service-role key (which already bypasses RLS), the public anon key is published to every browser. Any user can take that anon key from devtools and call the Supabase REST API directly to:
- `SELECT * FROM trade_sessions` → read every user's trade history (positions, P&L, broker, dates)
- `UPDATE user_plans SET plan='pro_yearly' WHERE user_id='self'` → free upgrade for everyone
- `SELECT code FROM coupons` → enumerate all beta coupon codes and their remaining uses
- `INSERT INTO trade_sessions (...)` → seed garbage rows
- `UPDATE/DELETE` arbitrary rows on every listed table.
**Why it matters:** This is total data exposure of every user's trade book and a free path to upgrade every account. This is the single biggest launch-blocking finding in this audit. The intent of the policies was clearly "service role can do anything" (which is already true by default), but as written they grant the same access to anonymous strangers.
**Suggested fix (implement before any production traffic):**
- Drop every `USING (true)` / `WITH CHECK (true)` policy listed above.
- Replace each with a role-scoped no-op or remove entirely. The service role bypasses RLS, so these policies serve no purpose for service-role traffic. Suggested rewrite:
  - `CREATE POLICY ... TO service_role USING (true)` (explicit and harmless), OR
  - just delete the policy and rely on service-role bypass.
- For tables that legitimately need anon-key access (none, in this codebase — all reads/writes go through API routes), write narrow policies like `USING (user_id = current_setting('request.jwt.claims', true)::json->>'sub')` — but **only after** confirming with `select * from auth.uid()` what value Clerk-mapped sessions actually expose.
- Verify each policy by connecting to Supabase with the anon key and attempting a `SELECT *` — it should return zero rows.
**Effort:** L (1-3 days — needs careful rewrite + manual verification per table)

#### F-016: `raw_files` SELECT policy admits "any anon user" through OR clause
**Severity:** BLOCKER
**Section:** 3
**Location:** `supabase/migrations/004_raw_files.sql:27-31`
**What I found:** Policy is:
```sql
CREATE POLICY "Users can view own files" ON raw_files
  FOR SELECT USING (
    user_id = current_setting('request.jwt.claims', true)::json->>'sub'
    OR anon_id IS NOT NULL
  );
```
The first branch is a no-op for Clerk users (the JWT claim is null since Clerk doesn't issue Supabase JWTs). The second branch — `anon_id IS NOT NULL` — is **true for every row that has any anon_id at all**, regardless of whose. So anyone with the anon key can SELECT every row in `raw_files` where `anon_id` was ever set, which includes all guest uploads.
**Why it matters:** Same exposure surface as F-015, scoped to `raw_files`. Trade-book PDFs and CSVs of guest users are readable by everyone.
**Suggested fix:**
- Replace with: `USING (anon_id = current_setting('app.anon_id', true) OR user_id = current_setting('request.jwt.claims', true)::json->>'sub')` — but verify the `set_config` for `app.anon_id` is actually issued per-request (it isn't today — the app uses service role).
- Simpler and safer: drop the policy, rely on service-role bypass via API routes only. Rewrite as `TO service_role USING (true)`.
**Effort:** S (< 2h — policy rewrite, but requires F-015 fix to be in motion)

#### F-018: Migration 009 schema for `payments` does not match production schema
**Severity:** HIGH
**Section:** 3
**Location:** `supabase/migrations/009_payments_table.sql` vs actual writes in `app/api/payments/create-order/route.ts:62-71` and `app/api/webhooks/razorpay/route.ts:75-90`
**What I found:** Migration 009 defines `payments` with columns `(id, user_id uuid NOT NULL, amount integer, currency, plan, provider, status, payment_id, created_at)`. The application code inserts rows with columns `(razorpay_order_id, clerk_id, email, razorpay_payment_id, razorpay_signature, completed_at, ...)` — none of these are in the migration. The header comment on 009 admits "this table already exists in production. Use CREATE TABLE IF NOT EXISTS to be safe."
**Why it matters:** (1) A fresh staging or future-dev database created purely from migrations gets a broken schema — every payment insert will silently 500 (and the catch block in `create-order` swallows it: "Failed to save payment to DB (continuing)"). The order succeeds at Razorpay, the user is charged, but no row is written. The webhook then can't find the order to update, plan upgrade is skipped, paid user remains free. (2) Violates the "migrations are the source of truth" principle. (3) The `user_id uuid` vs `clerk_id text` mismatch is the playbook's "Bug #5" — silent empty queries from column-type confusion.

#### F-024: Zero routes use schema validation (Zod, etc.) on `req.json()` / `req.formData()` bodies
**Severity:** HIGH
**Section:** 4
**Location:** every API route that destructures `req.json()` directly — 8 routes (`auth/sync`, `chat`, `coach`, `payments/create-order`, `payments/verify`, `trade-notes`, `coupons/redeem`, `analyse`)
**What I found:** Every JSON-body route does pattern: `const { x, y } = await req.json()`. No `zod`, `valibot`, `yup`, or any runtime validator in `package.json`. No type guards. Only `coupons/redeem` defends with `.catch(() => ({}))` against malformed JSON; the rest will throw and return a generic 500.
**Why it matters:** (1) Sending `{}` to most routes triggers `Cannot destructure property 'x' of undefined` — leaks stack traces in dev; in prod returns generic error but pollutes logs. (2) Sending wrong types (e.g. `plan: ['pro_yearly', 'pro_monthly']`) flows into the DB query and either errors loudly or silently corrupts. (3) Without a schema, your route contract is invisible to anyone reading the code — the only way to know what fields a route accepts is to grep call sites.
**Suggested fix:**
- Add `zod` to dependencies. (Consider `valibot` for smaller bundle.)
- Per route, define `const schema = z.object({...})` at top of file. Replace destructure with `const parsed = schema.safeParse(await req.json()); if (!parsed.success) return NextResponse.json({error: 'Invalid input', details: parsed.error.flatten()}, {status: 400})`.
- Start with the highest-blast-radius routes: `payments/create-order`, `payments/verify`, `coupons/redeem`, `auth/sync`, `coach`.
**Effort:** L (1-3 days for full sweep; M if you do top 5)

#### F-040: `avgWin` and `avgLoss` rounded to integer rupees, then used for R:R — loses precision
**Severity:** LOW
**Section:** 6
**Location:** `lib/kpi/computeKPIs.ts:161-167`
**What I found:** `avgWin = round(perTradeWinSum / winnersCount)`, `avgLoss = round(...)`. Then `riskReward = round(avgWin / avgLoss * 100) / 100`. For small-ticket option traders (₹50 wins / ₹40 losses), rounding to ₹50/₹40 then dividing distorts the ratio. Other rates (winRate, profitFactor) keep 1-2 decimal precision — inconsistency.

**Suggested fix:**

#### F-041: Date filtering uses local-tz `new Date(d)` while "today" comparison uses UTC `toISOString()` — IST off-by-one
**Severity:** HIGH
**Section:** 6
**Location:** `lib/kpi/computeKPIs.ts:200-220` (`filterByPeriod`)

#### F-056: No retry/backoff on 429/529/500 — Claude transient errors fail the user request
**Severity:** HIGH
**Section:** 8
**Location:** `app/api/analyse/route.ts:42-43`, all other Claude routes (none have retry)

The other 4 routes use `client.messages.create` (Anthropic SDK) which throws on non-2xx; the catch block returns 500 to the user. No exponential backoff anywhere. The audit pack §8.5 explicitly requires: 429 → backoff, 529 → retry, 500 → 2 retries then fail.
**Why it matters:** Anthropic 529 (overloaded) happens regularly during peak hours. A user hitting "Analyse" during a 529 spike gets a 500 error, retries manually, hits 529 again, abandons → support ticket. With one 1-second backoff and one retry, ~80% of 529s would self-heal silently.
**Suggested fix:**
- Create `lib/anthropicWithRetry.ts` wrapper:
  ```ts
  async function callWithRetry(client, params, maxRetries = 2) {
    for (let i = 0; i <= maxRetries; i++) {
      try { return await client.messages.create(params) }
      catch (e) {
        if (e.status === 429 || e.status === 529 || e.status >= 500) {
          if (i === maxRetries) throw e
          await sleep(1000 * Math.pow(2, i)) // 1s, 2s
        } else { throw e }
      }
    }
  }
  ```
- Replace direct `client.messages.create(...)` calls with `callWithRetry(client, ...)`.
- Same wrapper for the raw `fetch` calls in `analyse/route.ts:32` and `sessionSummarizer.ts:307`.
