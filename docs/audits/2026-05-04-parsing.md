## Observations from inventory alone (carried forward to Phase 5)

A few things that already look like real bugs *just from the file list and exports*, before any deep code reading:

1. **Two functions named `pairTrades`** in `lib/parsers/normalizer.ts` and `lib/parsers/pairTrades.ts` — same export name, different files. If both are imported anywhere, ambiguity. Need to verify which one wins at call sites.

2. **Three KPI calculators**: `lib/parsers/kpiCalculator.ts`, `lib/intake/kpiCalculator.ts`, `lib/kpi/computeKPIs.ts` — and `lib/compute/sessionMetrics.ts` also computes session-level KPIs. Four code paths can compute "win rate" — if formulas drift, dashboards disagree depending on which path renders.

3. **Two complete pattern engines**: `lib/analysis/patternDetector.ts` (832 LOC, "v4.0 legacy") and `lib/compute/patterns/*` (Module 2, ported from legacy). The bridge in `lib/analysis/module2Bridge.ts` includes a `buildShadowDiff` function — confirms parallel-running of both is being verified somewhere. Worth checking whether legacy or Module 2 is the actual production path right now.

4. **Two `parseTradeFile` exports** with different signatures: `lib/parsers/index.ts` (Buffer + filename) and `lib/parsers/universalParser.ts` (File object, client-side). Different return shapes. Easy to wire the wrong one at a call site.

5. **`raw_files` table has a `parser_version` column** but I don't see where it's written. If parsers evolve, we lose the ability to know which parser version produced a given saved row.

6. **`detected_currency` defaults to `'INR'`** in migration 002 — this is the india-only legacy assumption that the v5 landing redesign explicitly tries to undo. Worth checking whether parsers actually detect currency or just stamp 'INR' for everyone.

These are flagged as candidate findings for Phase 5; I haven't read the code yet.

### Bugs uncovered by the runner (N1-N9 — derived from per-fixture deltas)

These are the *new* findings the runner surfaced — not inventory observations.

#### N1 — IBKR signed-quantity not parsed *(severity: HIGH — silent total P&L loss)*

**File:** `lib/intake/rawExtractor.ts` and `lib/intake/tradePairer.ts`

**Symptom:** ibkr-csv and ibkr-xlsx fixtures both produced `netPnl: 0`, `wins: 0`, `losses: 0` — but `parsed: true`, `success: true`, no error. The user would see "0 P&L, 0 wins" with green checkmarks.

**Root cause:** IBKR exports use a **signed `Quantity` column** (positive = BUY, negative = SELL) instead of an explicit `BuySell`/`Side` column. The current parser detects only positive-quantity rows as legs, treats them all as `side: 'BUY'`, and pairing then has buys but no sells → all 6 trades come out as open positions with `entryPrice` and `pnl` zero. (Sample trade in `ibkr-csv.json:197` confirms: `side: "BUY", exitPrice: 0, pnl: 0` for AAPL despite the file containing both BUY 100 @ 182.50 *and* SELL 100 @ 184.20.)

**Fix:** In `rawExtractor.ts`, when no `side` column is mapped but a `qty`/`quantity` column has negative values, derive side from sign: `side: qty < 0 ? 'SELL' : 'BUY', qty: Math.abs(qty)`. The IBKR fixture also has an explicit `BuySell` column we could use as a tiebreaker, but the signed-quantity heuristic catches MT5/Tastyworks/etc. too. Test: ibkr-csv must produce netPnl ≈ +154.80.

#### N2 — Open-position trades tagged as "Winner" *(severity: HIGH — UI lies to user)*

**File:** `lib/intake/tradePairer.ts` (and `lib/parsers/normalizer.ts` line 145 `tag: hasPnl ? (t.pnl >= 0 ? 'win' : 'loss') : 'open'`)

**Symptom:** ibkr-csv produced 6 trades all tagged `tag: 'win', label: 'Winner'` despite `pnl: 0` and `exitPrice: 0`. These are open positions (no exit), not wins. ibkr-xlsx same — `side: 'SELL', tag: 'win', label: 'Winner', exitPrice: 0` (also a SELL→tagged-Winner contradiction).

**Root cause:** The default tag computation treats `pnl >= 0` as "win" without first checking whether the trade was actually closed. An open BUY with no SELL leg has pnl=0, which `>= 0` ⇒ "win".

**Fix:** Tag should be `tag: exitPrice === 0 ? 'open' : (pnl >= 0 ? 'win' : 'loss')`. (Robinhood handles this correctly with `tag: 'open'` — confirms the fix is local, not architectural.) Test: ibkr-csv all 6 trades must have `tag: 'open', label: 'Open Position'`.

#### N3 — Robinhood `Trans Code` column not recognised as side *(severity: HIGH — every Robinhood file silently broken)*

**File:** `lib/intake/rawExtractor.ts` synonym map / `lib/parsers/types.ts::COL.side`

**Symptom:** Robinhood CSV produced 12 trades all `side: 'BUY'`, all `pnl: 0`, all tagged `'open'`. The `Trans Code` column contains values like `BTO` (Buy To Open), `STC` (Sell To Close), `Buy`, `Sell` — the parser doesn't map "Trans Code" to side, so the field defaults and every row is BUY.

**Root cause:** `COL.side` regex (`lib/parsers/types.ts:105`) is `/^(side|trade.?type|buy.?sell|action|b.?s|direction|transaction.?type)/i`. "Trans Code" doesn't match any pattern. (`transaction.?type` would match "Transaction Type" but not "Trans Code".)

**Fix:** Add `trans.?code` to the side regex. Also add `BTO|STC|BTC|STO` to the side-value normalisation map (currently only handles BUY/SELL/B/S). Test: robinhood netPnl must reach +286.90.

#### N4 — Angel-One mis-detected as Zerodha *(severity: MEDIUM — wrong broker label, AI prompts wrong)*

**File:** `lib/config/brokers.ts`

**Symptom:** angel-one-tradebook.csv produced `broker: "Zerodha"`. The CSV contains no "zerodha" or "kite" string — but does have columns `Trade_ID` and `Order_ID`, both of which match Zerodha's keyword list. Phase-2 ≥2-keyword match triggers Zerodha.

**Root cause:** Zerodha's keyword list (`brokers.ts:24`) includes `'trade_id'` and `'order_id'` — generic terms that virtually every broker tradebook has. Phase-2 detection (>=2 matches) fires false-positive Zerodha for any file with `Trade_ID, Order_ID` columns. Also affects: Groww, Upstox, ICICI Direct, Kotak Securities, etc. — all use these column names.

**Fix:** Remove `'trade_id'` and `'order_id'` from Zerodha keywords (keep `'tradingsymbol'`, `'kite'`, `'order execution time'`, `'auction'` which are Zerodha-specific). Add Angel-One to `BROKER_REGISTRY` with keywords `['angel one', 'angelone', 'angel broking', 'AO-T-', 'AO-O-']` (last two are the angel-one trade-id prefixes). Same for Groww (`GR-O-`, `GR-T-`), Upstox (`UPL-`), ICICI Direct (`ICI-`), Kotak (`KOTAK-`).

#### N5 — Excel multi-sheet handling extracts wrong sheet *(severity: HIGH — IBKR XLSX silently mis-parsed)*

**File:** `lib/parsers/excelParser.ts`

**Symptom:** ibkr-xlsx-activity-statement.xlsx has 4 sheets: `[Account Information, Trades, Open Positions, NAV]`. Parser produced 6 trades all with `entryPrice: 0, exitPrice: 0`. The parsed headers field shows the **normalised mapping field names** (`["symbol","time","date","side","qty","price",...]`) — not the literal XLSX headers (`["Asset Category","Currency","Symbol","Date/Time","Quantity","T. Price",...]`). This suggests the parser merged/concatenated multiple sheets or read the wrong one.

Expected: 4 closed paired trades (AAPL, GOOG, META, EUR.USD) with realised P&L of +542.50. Actual: 0.

**Root cause:** `parseExcelBuffer` likely reads only sheet 0 (`Account Information`, which has no trade rows) OR navigates by content-detection without preferring the sheet literally named "Trades". Combined with bug N1 (signed-quantity), pairing fails entirely.

**Fix:** `parseExcelBuffer` should iterate sheets, prefer one named `/^trades$/i`, then fall back to first sheet whose headers look like trade columns (>= 4 columns matching `COL.*` regex). Also, extracted `headers` should be the **literal** column names, not the mapped names — current behaviour makes debugging harder. Test: ibkr-xlsx must show `headers: ["Asset Category", "Currency", "Symbol", "Date/Time", ...]` and `netPnl ≈ +532.50`.

#### N6 — FIFO partial-fill produces extra "trades" *(severity: MEDIUM — trade count inflated)*

**Symptom:** zerodha-equity (5 closed + 1 open expected, 8 actual), upstox (5 closed expected, 7 actual), groww (5 closed + 1 open expected, 7 actual), angel-one (5 closed expected, 7 actual). NetPnl sums correctly (since FIFO matches sum to the same total) but trade COUNT and WIN-RATE are inflated.

**Root cause:** `lib/parsers/normalizer.ts::pairTrades` does FIFO matching across `(symbol+date)`. When a buy has `qty=300` and there are two sells of `200+100`, FIFO produces TWO paired trades instead of one logical trade. Same for averaging-down clusters (3 buys of 100 each → 1 sell of 300 = 3 paired trades for one round-trip). The expected.json fixtures count 1 round-trip per symbol-day, the parser counts each fill match.

**Fix:** Two reasonable behaviours. Either (a) keep FIFO output as-is and document that "trade count" = "fill match count", OR (b) consolidate same-day same-symbol same-side groups before pairing — averaged buy price × averaged sell price → 1 paired trade. Option (b) matches what most users mentally count as "a trade". P&L is identical either way; UX is the difference. Recommend (b) for Indian retail context (intraday averaging is common).

#### N7 — `IntakeKPIs.netPnl` is GROSS, not NET *(severity: MEDIUM — naming lies)*

**File:** `lib/intake/kpiCalculator.ts:24` (`const netPnl = grossProfit + grossLoss`)

**Symptom:** Field is named `netPnl` but it is the sum of per-trade `pnl` (which itself is gross of fees in most fixtures). MT4 fixture: actual `netPnl: 25.53`, expected `grossPnl: 25.53` ✓ but expected `netPnl: 1.58` (after $-21.95 commission+swap). The intake KPI's "net" never subtracted the `totalFees` field that exists right next to it.

**Root cause:** `calculateIntakeKPIs` computes `netPnl = grossProfit + grossLoss` (line 24). It exposes `totalFees` (line 28) but never subtracts it. The dashboard then displays `netPnl` as "Net P&L" without the user knowing fees aren't deducted.

**Fix:** Either (a) actually subtract fees: `netPnl = grossProfit + grossLoss - totalFees` and rename `grossPnl = grossProfit + grossLoss`, OR (b) rename `netPnl → grossPnl` and add a true `netPnl = grossPnl - totalFees`. Option (b) is safer (no surprise behaviour change); option (a) matches the field's name. Recommend (b) — also requires UI rename across all dashboards.

#### N8 — Forex P&L vs ticker price triggers false validation warnings *(severity: LOW — noise)*

**File:** `lib/intake/tradeValidator.ts`

**Symptom:** mt4 fixture produced 7 validation warnings of the form `"Trade 0: P&L (200) exceeds 10x trade value (1.0872)"`. The validator is comparing P&L (in account currency, USD) against trade value (price × qty, where price is the EUR/USD pair quote ≈ 1.08). This is a forex-specific calculation — for EURUSD with size 1.00 lot, the "trade value" is 100,000 base currency units, not the 1.08 quoted price.

**Fix:** Either (a) skip the 10x-trade-value validator for forex fixtures (detected market === 'Forex'), or (b) multiply by lot size when known (1.00 lot = 100,000 base units). Option (a) is simplest. Test: mt4 should show 0 warnings.

#### N9 — Robinhood market detection returns "Unknown" despite NYSE tickers *(severity: LOW — currency fallback risk)*

**File:** `lib/parsers/types.ts::detectMarket`

**Symptom:** robinhood-account-statement.csv contains AAPL, TSLA, AMD, SPY, QQQ — all NYSE/NASDAQ tickers. The market detector returned `Unknown` because the file text doesn't contain `nyse`, `nasdaq`, or any of the explicit US-broker keywords (`fidelity`, `schwab`, `robinhood`, etc.). Wait — `robinhood` IS in the keyword list (`types.ts:136`), but the broker detector caught it first and may have early-returned. Worth tracing.

Actual impact: combines with **Finding F** (currency fallback). Robinhood market=Unknown → no signal to override the `|| 'INR'` default → US user sees ₹ in dashboard despite being on Robinhood.

**Fix:** When `broker === 'Robinhood'` (or any US-only broker), force `market: 'NYSE', currency: 'USD'`. Same for Schwab/Fidelity/E*Trade/Webull. Already partly there in `lib/config/brokers.ts` registry — extend `BROKER_REGISTRY` entries with `defaultMarket` and `defaultCurrency` fields, and have `extractRawFile` use them as overrides when detection returns Unknown.

| KPI | Canonical | Calc 1 (intake) | Calc 2 (parsers) | Calc 3 (dashboard) | Calc 4 (Module 2) | Edge case that would diverge |

### Q1 — KPI rounding contract *(raised by B.4)*

The repo has no formal rounding contract. The four calculators imply one by majority vote:

| Quantity | intake | parsers | dashboard | sessionMetrics | Implied contract |
|---|---|---|---|---|---|
| `profitFactor` | round 2dp | round 2dp | round 2dp | full precision | **2dp** (3-of-4) |
| `winRate` (%) | round 2dp | round 2dp | round 1dp | full precision (×100 by caller) | **2dp** (2-of-4 strict) |
| `avgWin` / `avgLoss` (currency) | round 2dp | round 2dp | round to integer | full precision | **2dp** (2-of-4 strict) |

### Q3 — Trade-count semantics *(raised by N6)*

When a user does 1 BUY of 300 and 2 SELLs of 200+100, do we report 1 trade or 2?

(a) **1 trade** — averaged. Matches retail mental model ("I traded SBI today, it went my way"). Currently NOT what the parser does (FIFO produces 2). Win rate becomes more meaningful (1 trade, 1 win).

(b) **2 trades** — FIFO match count. Matches what the broker statement literally shows. Currently what the parser does. Win rate is fill-weighted, not trade-weighted.

The fixtures' `expected.json` files universally count 1-per-round-trip (option a). Production code does option b. **Need a decision before N6 fix.**

### Q4 — `IntakeKPIs.netPnl` semantics *(raised by N7)*
